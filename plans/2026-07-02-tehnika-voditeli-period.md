# План: диапазон дат для "Техника" и "Водители" + единый источник данных

**Дата:** 2026-07-02
**Статус:** в работе

## Проблема

После перехода на июль обнаружился рассинхрон на странице "Водители": колонка "Заказов"
уже показывала июль (блок заказов умеет копить историю по месяцам), а "План/Факт/%" -
всё ещё июнь (лист "ТОП_водителей_по_плану" обновляется 1С только на текущий месяц,
истории нет). Такая же проблема в "Технике" - "Нормализованные_данные" тоже без истории.

Заодно запрошено: возможность выбрать диапазон дат (не только текущий месяц) на обеих
страницах, чтобы смотреть история за 2+ месяца сразу. И убрать блок "Топ-10 водителей
по выполнению плана" (график) - не нужен.

## Решение

**Один источник данных вместо двух.** Лист "История_финансов" уже пишет подневный снимок
по каждой машине 6 раз в день (см. saveFinancialHistory() + setupTrigger(), часы 10-20 по
Москве) - и уже сейчас содержит достаточно истории, чтобы восстановить прошлые месяцы.
Значения в снимке - **накопительный итог с начала месяца** (как в Нормализованные_данные,
которые 1С каждый месяц обнуляет), не дельта за день.

И "Техника" (сейчас getVehiclesData() из Нормализованные_данные), и "Водители" (сейчас
getDriversData() из отдельного листа ТОП_водителей_по_плану) переводятся на один и тот же
агрегатор поверх История_финансов. "Водители" перестаёт быть отдельным листом - строится
из тех же машин (водитель через Штатку), План=v.plan, Факт=v.revenue.

Это одновременно чинит рассинхрон (один источник - разъехаться нечему) и даёт историю
по месяцам бесплатно, без ожидания 1С (в отличие от заказов и "Данные_1С").

## Логика агрегации по диапазону дат

Для диапазона [from, to], который может пересекать границу месяца:
1. Разбить диапазон на месяцы, которые он затрагивает.
2. Для каждой (машина, месяц) - взять снимок с САМОЙ ПОЗДНЕЙ датой из диапазона внутри
   этого месяца (снимок кумулятивный, брать последний = взять "итог на конец периода").
3. Просуммировать эти по-месячные итоги (revenue, fot, fuel, parts, fines, tolls, profit,
   plan) по всем затронутым месяцам для этой машины.

Отсюда и "планы множатся на количество месяцев" (Влад, 2026-07-02) - выбрал 2 месяца,
план = план июня + план июля, это ожидаемо, не баг.

Тип/статус/марка/водитель - не суммируются, берутся из самого свежего снимка в диапазоне
(статус) и из Штатки (марка/водитель/тип - "текущее состояние", как и в остальном проекте).

Частный случай: диапазон = только текущий месяц, работает так же (один месяц в сумме) -
поэтому дефолтный вид (без выбора периода) тоже переводим на этот агрегатор, а не только
"исторический" режим - чтобы Техника и Водители физически не могли разъехаться.

## Риски

- История_финансов может иметь пропуски (если раннер не запускался несколько дней подряд -
  маловероятно, но не исключено) - "последний снимок в месяце" в таком случае может быть
  не последним днём месяца, а раньше. Не страшно для агрегации (данные кумулятивные, просто
  чуть не до конца месяца), но стоит проверить на реальных данных перед релизом.
- Первый прогон агрегатора - обязательно сверить с текущими live-числами на Панели
  (Нормализованные_данные) за сегодня, они должны почти совпадать (расхождение - в пределах
  времени с последнего triggers-запуска, до ~2 часов).
