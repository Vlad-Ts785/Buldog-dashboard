## Что сделали за сессию

### Личная страница менеджера — доработка и редизайн
- 4 равных блока в ряд (вместо 4 разных по размеру): Заработал / Документы / Дебиторка / Заказы по технике
- «Документы»: сегментированная полоса + точные цифры по каждой стадии (нет путёвки/не проведено/не закрыта) вместо абстрактных цветов
- Широкая шкала «Выполнение плана продаж»: факт + прогноз по темпу (факт/день×дней_в_месяце) + статус текстом
- «Заработал»: прогресс до следующей ступени оклада + прогноз зарплаты (через `calcMgr` на масштабированных по темпу цифрах — без дублирования формулы)
- «Заказы по технике»: тралы/длинномеры, две шкалы друг относительно друга
- Большой блок «Дебиторская задолженность»: по контрагенту, со сроком просрочки от даты **первого неоплаченного заказа** этого клиента (новое поле `first_unpaid_date` в бэкенде)
- Каждый редизайн сначала показывался превью (`mcp__visualize`) перед переносом в реальный код — подтвердилось как рабочий процесс

### Новая формула найма с 1 июля 2026
- 8% менеджеру / 8% логисту-исполнителю / 2% его руководителю / 2% Рыщанову от маржи найма — **только если маржа по заказу ≥23%**, иначе 0
- Прус-Роскошный с июля — только логист (500₽/заказ + 8% маржи), убран из списка менеджеров
- Period-gated: `isNewRulesPeriod()` сравнивает `D.orders.period` с `'2026-07'` — старые периоды считаются как раньше, без изменений
- Перед кодом — уточняющие вопросы Владу (3 раунда), т.к. формула напрямую про деньги людей; зафиксировано точно с его слов

### Выбор периода (Заказы/Менеджеры/Логисты/Зарплата)
- Бэкенд: `normalizeOrders()`/`getOrdersData()` разбиты на чистые функции (`parseOrdersRawRows`, `aggregateOrdersRows`) — переиспользуются и для текущего месяца, и для архивов на лету
- Новые `doGet`-экшены: `available_periods`, `orders_period`
- ВП своего парка за прошлый период — из `История_финансов` (подтвердил Влад)
- Фронтенд: подмена `D.orders`/`D.summary.profit` на время синхронного рендера 4 вкладок, с возвратом обратно — остальные вкладки не задеты

### Фикс: поздние коррекции прошлого месяца
- `archiveOrdersIfNeeded()` раньше архивировала "текущий" месяц при любом несовпадении периода, не проверяя направление
- Если бухгалтерия дошлёт исправленный июнь уже в июле (после первого июльского отчёта) — скрипт чуть не затёр живые данные старыми
- Теперь: вперёд по календарю — архивируем и переключаемся как раньше; назад — только обновляем архив прошлого месяца, живую таблицу не трогаем

### Вход через Google + роли admin/manager
- Полный проход через настройку: OAuth Client ID в Google Cloud Console (с нуля, пошагово, т.к. Влад впервые это делал)
- Бэкенд: `verifyGoogleToken_()` (tokeninfo), лист «Доступ» (email→имя→роль), `getManagerView_()` — урезанный ответ для роли manager (только свои строки, без доступа к данным компании/других менеджеров)
- Фронтенд: экран входа, `apiUrl()` хелпер прикрепляет токен ко всем запросам, роль `manager` прячет весь сайдбар кроме личной страницы
- Баг и фикс: `?onload=` у скрипта Google в `<head async defer>` состоял в гонке состояний с нашим кодом — тихо не срабатывал без единой ошибки в консоли. Перенесли скрипт в конец `body` без async/defer, вызов явный
- Второй баг и фикс: после входа интерфейс не обновлялся сам (только после ручного refresh) — вместо разбора точной причины (особенности браузера сразу после закрытия попапа входа) просто перезагружаем страницу программно тем же путём, который уже надёжно работал

### Публикация на GitHub Pages
- Нашли: `origin/main` содержал один несвязанный коммит ("Add files via upload", 20 июня) — старый калькулятор, не наш дашборд
- Влад параллельно сам залил через веб-интерфейс GitHub ещё один файл в новую ветку `Vlad-Ts785-patch-1` — и Pages переключился на неё (более старая версия, без личной страницы/входа)
- Решение: запушили актуальную ветку, `force-push` в `origin/main` (после явного подтверждения), переключили Pages Source обратно на `main` руками в настройках
- Сайт `vlad-ts785.github.io/Buldog-dashboard` теперь реально отдаёт актуальный код со входом через Google — проверено напрямую curl'ом (не только на глаз)

## Что можно было лучше

- Несколько раз не давал превью разработанного UI **до** переноса в код в начале сессии (вкладка KPI-карточек) — после явной просьбы Влада «покажи превью прежде чем запускать» стало системным шагом для всех последующих визуальных решений. Стоило сделать это правилом с самого начала, не дожидаясь просьбы.
- Дважды проскочил английский в коротких репликах-нарративах вокруг tool calls, несмотря на правило «только русский» — оказалось, проблема была глубже видимого текста (сам процесс рассуждения тоже куда-то просачивается в интерфейс). Поправил память дважды, второй раз — точнее.
- При первой попытке push в main едва не пропустил, что GitHub Pages раздаёт корень репо, а не `files/` — нашёл только проверкой curl'ом после публикации, не заранее. Стоило проверить настройки Pages (или хотя бы реальный live-контент) до, а не после первого деплоя.

## Как было / как стало

- **Было:** дашборд открыт всем по прямому URL без единой проверки, GitHub Pages показывал случайный устаревший файл, личная страница менеджера — общий вид без авторизации, зарплата найма по старым правилам, нет архивной истории заказов.
- **Стало:** вход только через Google + email-список, роль `manager` видит только себя, сайт реально опубликован и актуален, зарплата найма с 1 июля считается по новым правилам автоматически (старые периоды — по-старому), можно посмотреть любой прошлый месяц с архивными данными на 4 вкладках.
