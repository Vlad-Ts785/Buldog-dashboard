# План: Штатка как рабочий инструмент дашборда

**Дата:** 2026-07-01
**Статус:** в работе

## Цель

Влад каждое утро звонит на Базу, опрашивает по каждой машине статус и вручную вбивает в
Google Таблицу (сетка по дням месяца). Хочет делать это прямо в дашборде вместо Таблиц.
Хочет, чтобы история по месяцам не терялась (сейчас теряется - см. Контекст).

## Контекст (изучено на реальном файле Штатки, 2026-07-01)

Штатка - самый нагруженный лист в системе, его читают `getStaffData()` -> `getFleetStatus()`,
`getVehiclesData()` (водители), `normalizeReport()` (джойн по госномеру), `saveFinancialHistory()`,
`getRepairsData()`. Менять нужно осторожно.

**Реальная структура (не совпадает с "там ничего сложного"):**
- Строка 5 - заголовки, данные с 6-й. ~176 непустых строк по факту (тралы + длинномеры).
- Дни месяца - это КОЛОНКИ ("01.06." … "30.06."). Позиция колонки "Статус на сегодня"
  СДВИГАЕТСЯ в зависимости от числа дней в месяце (30 vs 31) - код уже ищет её по названию,
  не по букве, новый код должен делать так же.
- В клетках дневной сетки сейчас **только числа** (0-5, заказы за день) - букв не было.
- В том же листе вперемешку: тралы/длинномеры (нужны), трейлеры без тягача ("перецепка" -
  реальное текущее состояние парка, не мусор), легковые машины сотрудников (НЕ нужны),
  отдельный список ~50 водителей внизу листа (НЕ нужен).
- Сверху (строки 1-4) - сводка план/факт, дублирует наш дашборд.
- **История НЕ сохраняется автоматически.** Влад руками копирует июнь в отдельную таблицу
  перед сбросом на новый месяц - при масштабировании это перестанет работать нормально.
  На момент изучения (01.07, "сегодня" по датам в проекте) сетка ещё показывает июнь
  (01.06.-30.06.) - переключение на июль ещё не произошло, значит миграцию можно сделать
  из уже прочитанного файла, без запроса отдельной июньской копии.

## Область (Фаза 1)

**Работаем только с парком грузовиков** (тралы + длинномеры). Легковые машины, список
водителей - НЕ трогаем.

**Статусы дня (один статус на клетку, никогда два одновременно):**
- Число 0-5 - количество заказов за день (0 = "без заказа")
- `Р` - ремонт
- `В` - без водителя (сюда же входит "приёмка" - один день, когда заступает новый/вахтовик,
  отдельного статуса под это не делаем по просьбе Влада). **Не `Б`** - в реальном выпадающем
  списке "Тралы" в оригинальной Штатке используется именно буква `В` (уточнили 2026-07-04,
  когда Влад прислал скриншот настройки списка) - код исправлен под неё.
- `РВ` - **новый статус** (ремонт + без водителя одновременно) - Влад попросил явно, база
  постоянно спорит, что ставить, когда оба условия верны сразу. Само сокращение теперь
  логично читается как "Р"+"В" (ремонт + без водителя).

**Перецепка (перекомплектация тягач+прицеп)** - Фаза 2, не в этой итерации.
**"Пересадить водителя"** - тоже Фаза 2.

## Решение по хранению

Тот же приём, что уже сработал для заказов: вместо "широкой" сетки (день = колонка,
перезатирается каждый месяц) - **новый лист "Штатка_история"** в длинном формате:
`Дата | Госномер | Статус`. Одна строка = один день одной машины. Копится бесконечно,
показ конкретного месяца - просто фильтр по дате (как уже делаем для заказов).

- [x] Новый лист `Штатка_история` (Дата | Госномер | Статус)
- [x] `migrateShtatkaGridToHistory(year)` - разовая миграция: читает текущую сетку Штатки
      (динамический поиск дневных колонок по паттерну "ДД.ММ."), пишет по строке на
      непустую клетку. Идемпотентно (сначала чистит целевой диапазон дат, потом пишет).
      Проверено на РЕАЛЬНЫХ данных Штатки (Node, без Apps Script) - 30 июньских колонок
      найдены точно, "без тягача" корректно исключены.
- [x] `getShtatkaGridData(ss, monthKey)` - читает список машин из Штатки (тип/тягач/прицеп/
      водитель) + статусы за месяц из Штатка_история, собирает в JSON для дашборда
- [x] `setShtatkaStatus(ss, gos, date, status)` - валидирует статус (0-5 или Р/Б/РВ),
      пишет/обновляет одну строку в Штатка_история

### Фаза 2c. Смена курса - Штатка в вебе становится витриной, не рабочим инструментом (2026-07-04)

Дашборд теперь просто автоматически показывает историю - ручной ввод в вебе остаётся
как опция на случай отсутствия ноутбука с таблицей, но не основной способ.
- [x] Буква статуса "без водителя" исправлена с `Б` на `В` (реальная буква в списке базы -

## Риски

- Штатку читают 5+ функций дашборда - миграция и новый код не должны трогать существующие
  колонки/структуру, только добавлять параллельный источник (Штатка_история).
- Нет доступа к боевому Apps Script для прогона - весь новый код проверяется через Node
  (синтаксис + логика на синтетических данных), как и остальной код этой сессии.
