# Дашборд ЯРД - навигация

Карта проекта. Читай в начале каждой сессии.

## Что это за проект

**Дашборд ЯРД** - внутренняя BI-система для директора ООО «Бульдог» (группа ЯРД Империал).
Сводит данные из 1С, Google Sheets, Роснефть и телефонии в одно окно.
Влад открывает ноутбук - за 2 минуты видит состояние бизнеса без звонков сотрудникам.

Подробности - `.business/INDEX.md`.

## Порядок чтения в начале любой сессии

1. `../ai-clone/INDEX.md` - кто такой Влад и как он думает
2. `.business/INDEX.md` - контекст компании, команда, сегменты ЦА
3. `CLAUDE.md` (этот файл) - стек, правила, навигация
4. Зона задачи: нужный файл из `.business/`, `plans/<актуальный план>`, или код

## Архитектура данных

```
1С -> Email (1С шлёт отчёты)
       |
  Apps Script (парсит Gmail)
       |
  Google Sheets (20+ листов)
       |
  doGet() JSON API
  /           \
Дашборд      Telegram-бот
(GitHub)     (3 сообщения)
```

**Ключевые листы:** Штатка (ЖИВОЙ документ - статусы машин), Нормализованные_данные,
Менеджеры_данные, История_показателей, История_финансов, Заказы_данные (+ архивы
Заказы_YYYY-MM), Планы_менеджеров, ДЗ_данные + История_ДЗ (дебиторка, отчёт 1С раз в
день в 15:00, см. `plans/2026-07-08-debt-receivables-tab.md`).

**Штатка - критические колонки:** A=тип, C=госномер, AF=статус машины,
AG2/3/4=тралы, AH2/3/4=длинномеры, AJ2/AK2=без заказа.

## 8 правил репо

1. **Не хардкодить** токены, ID, константы. Всё в `.env` или конфиге.
2. **Не деплоить без явной команды.** Push в main = автодеплой на GitHub Pages.
3. **Удалять одноразовый код** после использования.
4. **`.env` только через `grep "^VAR=" .env`** - никогда не читать целиком.
5. **Большая фича = план в `plans/`.** Граница: задача больше часа - пишем план.
6. **Цикл после каждой фичи:** найди ошибки -> что упустили -> security review.
7. **`git add` поимённо.** Никогда `git add .` - параллельно работают другие чаты.
8. **Ретроспектива в конце сессии** - `retrospectives/YYYY-MM-DD_HH-MM_название.md`.

## Цикл после каждой фичи (обязательно перед закрытием задачи)

1. Найди ошибки в написанном коде - исправь
2. Что упустили (граничный кейс, валидация, ошибки) - исправь
3. Это реально решает нашу задачу? Сверь с `.business/` и `plans/`
4. Security review: ПДн, внешние запросы, токены в коде

## Любая ошибка -> правило в `../ai-clone/feedback/`

Когда Влад говорит «не так, потому что...» - оформи файл в `../ai-clone/feedback/`:
- **Rule**: что делать / не делать
- **Why**: причина, прецедент
- **How to apply**: когда срабатывает

## Безопасность

- `.env` не читать целиком. Используй `grep "^VAR=" .env`.
- Токен Telegram - только в `.env`, не в коде и не в CLAUDE.md.
- `rm -rf` заблокировано в `settings.json`. Перед удалением - показывай список.
- Скиллы и MCP - только после аудита безопасности.

**Вручную через браузер (запасной путь)** - Развернуть -> Управление развёртываниями ->
Новая версия -> Развернуть.

**В обоих случаях** - `clasp push`/ручное сохранение только обновляет код редактора. Если
менялась функция вроде `setupTrigger()`, которая должна быть ВЫПОЛНЕНА (не только
задеплоена), это отдельный шаг - попроси Влада запустить её вручную в редакторе (Выполнить
-> выбрать функцию -> Выполнить), скрипты не умеют вызывать сами себя через clasp.

**Координация между сессиями (ОБЯЗАТЕЛЬНО)** - параллельно могут работать другие чаты и
деплоить туда напрямую, минуя git. Уже дважды приводило к тихому откату чужого фикса
(версии 90-92, см. `scripts/DEPLOY_LOG.md`). Перед КАЖДЫМ `clasp push`:
1. `clasp pull` в `Дашборд_Apps_Script`.
2. `diff --strip-trailing-cr` с `scripts/full_script_final.js` из git HEAD.
3. Если diff непустой - в живом файле есть чужая недокоммiченная работа, сначала перенеси
   её в репозиторий отдельным sync-коммитом, и только потом свои правки.
4. После `push`+`create-version`+`redeploy` - добавь строку в `scripts/DEPLOY_LOG.md`.
