# План: выбор периода на вкладках "Продажи" (Заказы/Менеджеры/Логисты/Зарплата)

**Дата:** 2026-06-30
**Автор:** Клод (с Владом)
**Статус:** завершён

## Цель

Дать Владу выбрать любой прошлый месяц на 4 вкладках раздела "Продажи" (Обзор заказов,
По менеджерам, По логистам, Зарплата) и увидеть сохранённые данные за этот месяц -
без перезаписи того, что видят остальные вкладки (Панель, Система решений и т.д. - всегда
текущий месяц).

## Контекст

Архивы заказов (`Заказы_YYYY-MM`) уже сохраняются автоматически (см. фикс с поздними
коррекциями, 2026-06-30) - в сыром виде. Сейчас разбор и агрегация жёстко привязаны
к текущему месяцу (`Заказы_данные`). Зарплата Рыщанова считается от валовой прибыли
своего парка - Влад подтвердил, что для прошлых периодов её можно брать из
`История_финансов` (там есть посуточные снимки накопительного ВП).

Все 4 вкладки уже читают данные из единого `D.orders` (и зарплата - ещё из
`D.summary.profit` для Рыщанова) - это позволяет переиспользовать существующий
рендер без переписывания, просто подменяя источник данных.

## Фазы

### Фаза 1. Бэкенд: чистые функции разбора и агрегации (~40 мин)
- [ ] `normalizeOrders()` → выделить `parseOrdersRawRows(rawData)` (сырые строки -> нормализованные,
      без записи в лист)
- [ ] `getOrdersData(ss)` → выделить `aggregateOrdersRows(rows)` (нормализованные строки -> JSON,
      без чтения листа)
- [ ] Оба места переиспользуют чистые функции, поведение для текущего месяца не меняется

### Фаза 2. Бэкенд: данные за период (~40 мин)
- [ ] `getOrdersDataForPeriod(ss, period)` - читает архив `Заказы_YYYY-MM`, прогоняет через
      parseOrdersRawRows + aggregateOrdersRows, отдаёт тот же формат, что и текущий месяц
- [ ] `getGrossProfitForPeriod(ss, period)` - читает `История_финансов`, берёт последнюю дату
      внутри периода, суммирует ВП по всем машинам на эту дату
- [ ] `doGet`: `action=orders_period&period=YYYY-MM` -> `{ orders, summary: { profit } }`
- [ ] `doGet`: `action=available_periods` -> список месяцев, по которым есть архив

### Фаза 3. Фронтенд: переключатель периода (~50 мин)
- [ ] Выпадающий список в `page-header`, виден только на вкладках sales/sales-mgr/sales-log/sales-salary
- [ ] `changePeriod(period)`: фетчит `orders_period`, кладёт в `periodOrdersData`/`periodGrossProfit`
- [ ] `renderSalesTabsForPeriod()`: подменяет `D.orders`/`D.summary.profit` на период, синхронно
      вызывает существующие renderSales/renderSalesMgr/renderSalesLog/renderSalesSalary,
      сразу возвращает обратно - остальные вкладки не задеты
- [ ] `render()`: заменить прямые вызовы 4 функций на вызов `renderSalesTabsForPeriod()`
- [ ] Список периодов грузится один раз при первом входе на одну из 4 вкладок (как `loadHistoryData()`
      для Динамики)

## Критерии готовности

- [ ] Выбор прошлого месяца на вкладке "Зарплата" показывает прошлые цифры, без ошибок в консоли
- [ ] Переход на "Панель" при выбранном прошлом периоде показывает текущий месяц как обычно
- [ ] Возврат на "Текущий месяц" в селекторе восстанавливает живые данные
- [ ] Рыщанов: ВП за прошлый период берётся из Истории_финансов, не из текущего месяца
- [ ] Список периодов не содержит месяцев без архива

## Риски

- Если "Отчёт парк" (источник Истории_финансов) тоже имеет лаг коррекций в начале месяца,
  как заказы - первые дни нового месяца историческое ВП может быть не точным. Не чиню сейчас,
  Влад не поднимал этот риск для парка отдельно - отметить на будущее.
- Производительность: разбор архива на лету при каждом выборе периода - для масштаба заказов
  одного месяца (сотни-тысячи строк) укладывается в лимиты Apps Script, не кешируется.

## Открытые вопросы

- Период действует на все 4 вкладки сразу (общий выбор для раздела "Продажи"), а не отдельно
  на каждую - решение по умолчанию, спросить Влада если не так.
