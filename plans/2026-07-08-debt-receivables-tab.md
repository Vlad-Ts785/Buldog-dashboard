# План: вкладка "ДЗ" (дебиторская задолженность)

**Дата:** 2026-07-08
**Автор:** Клод
**Статус:** в работе (Фазы 1-4 реализованы, ждёт первого реального импорта и проверки Владом)

## Цель

Новая вкладка на дашборде с реальной дебиторской задолженностью (кто должен, сколько,
как давно) из официального отчёта 1С, вместо приблизительной оценки "должники"
(`сумма заказа - оплата`), которая уже отмечена в памяти как неточная.

## Контекст

Влад настроил в 1С ежедневную (15:00) отправку письма "Отчет по дебиторской
задолженности тралы от DD.MM.YYYY" с Excel-вложением на ту же почту, которую уже
парсит Apps Script (`v.tsutsurin@yard-imperial.ru`).

Разобрал присланный образец файла (`Дебиторская задолженность (Организация Контрагент) -
копия (XLSX).xlsx`, 5538 строк) - это выгрузка "Взаиморасчёты" из 1С, три уровня
вложенности: **Юрлицо → Контрагент → Документ**.

Влад подтвердил ключевые решения по фильтрации (2026-07-08, устно):

1. **Не фильтровать по юрлицу.** Часть клиентов до сих пор ведётся не через "Бульдог",
   а через другие компании группы (например, тендер выигран на "Ярд Империал" - работать
   так минимум ещё год, пока не переведут всё на "Бульдог"). Проверено на самой выгрузке:
   во всех 5 юрлицах присутствуют одни и те же отделы "Тралы" - значит трал-бизнес реально
   размазан по юрлицам, фильтр по юрлицу отрезал бы часть настоящей ДЗ.
2. **Фильтровать по менеджеру** (список = TRAL_MANAGERS, как везде на дашборде). 1С уже
   прислал файл предфильтрованным по нужным менеджерам - но фильтруем ещё раз на своей
   стороне (defensively), на случай если Влад в 1С поменяет отбор.
3. **"master"** (генерик-пользователь на крупных строках "Поступление...") - проверено:
   у этих строк всегда пусто в "Сумма долга" (только "Сумма нашего долга" - незаработанные
   авансы). В агрегации по клиентской ДЗ они естественно дают 0, отдельно исключать не надо.
4. **Убрать телефон из имени менеджера.** В 1С часть менеджеров записана с телефоном
   ("Савиток Олеся Анатольевна 8-985-150-11-85") - Влад: "это вообще нужно везде
   исключать... у менеджера есть фамилия имя отчество, и всё, больше ничего не надо".
   На фронтенде уже есть `cleanName()` для этого - но не везде применяется. Разово
   проверить остальные вкладки на утечку телефона (отдельный маленький фикс).
5. **"Гарантийный депозит"/"Гарантийный платёж"** - НЕ часть клиентского долга. Это
   зарплатный механизм: если клиент не платит, депозит удерживают из зарплаты
   ответственного менеджера как гарантию; когда клиент платит - депозит возвращают
   менеджеру. Показывать отдельно как контекст, не смешивать с "клиент должен нам".
6. **Главная цифра ДЗ = "Сумма долга" минус "Сумма аванса"** (просто чистый баланс без
   гарантийных колонок - те за скобками, см. п.5). Более сложный нетто из отчёта ("Итог")
   намеренно не используем как основную цифру - Влад подтвердил в общих чертах ("красным
   с отрицательными значениями = реальный долг"), но не давал точной формулы; начинаем с
   простой и понятной, уточняем по ходу ("начнём, посмотрим что получится").

**Даты для расчёта "возраст долга":** в названии каждого документа есть дата
("Реализация (акт...) 00БП-004941 **от 30.06.2026** 23:59:59") - 100% строк
(1472 из 1472 в образце) содержат распознаваемую дату, можно считать дни просрочки.

**План на будущее (не в этой итерации):** если алгоритм расчёта ДЗ окажется надёжным -
рассмотреть замену текущего приблизительного расчёта "должников" на "Личной странице"/
"Системе решений" на данные из этой новой вкладки. Отдельная задача, после того как
Влад проверит вкладку ДЗ на реальных данных.

## Структура источника (для парсинга)

Лист "Лист_1", данные с строки 10 (после параметров/заголовков), последняя строка -
"Итого". Три типа строк по колонке A + наличие колонки D:

- **Юрлицо**: `A` начинается с "NN. " (например "08. БУЛЬДОГ ООО"), колонки D/F пустые.
- **Контрагент**: `A` = название клиента, колонки D/F пустые, `G` = сумма долга контрагента.
- **Документ**: `D` (Отдел) и `F` (Менеджер, часто с телефоном) заполнены, `A` = название
  документа с датой внутри ("Реализация...", "Таблица заказов...", "Поступление...").

Колонки (индекс с 1): A=Организация/Контрагент/Документ, D=Отдел, F=Менеджер,
G=Сумма долга, H=Сумма аванса, I=Гарантийный платёж, J=Гарантийный депозит, K=Итог,
L=Сумма нашего долга, M=Сумма наших авансов, N=Общий итог.

## Фазы

### Фаза 1. Backend: импорт и нормализация (~2ч)
- [ ] `importDebtReport()` - по аналогии с `importOrdersReport()`/`importParkReports()`:
      ищет письмо `subject:"Отчет по дебиторской задолженности" has:attachment newer_than:2d`,
      конвертирует Excel во временный Google Sheet, читает.
- [ ] `parseDebtRawRows_(rawData)` - разбирает 3-уровневую иерархию в плоский список
      `{contragent, manager, org, docDate, docDesc, debt, advance, guaranteePayment,
      guaranteeDeposit}`. Дата документа - regex `от (\d{2}\.\d{2}\.\d{4})` из `docDesc`.
      Менеджер - через ту же логику, что фронтендный `cleanName()` (стрип телефона).
      Фильтр по TRAL_MANAGERS на менеджере (defensively, см. п.2 контекста).
- [ ] Пишем в новый лист "ДЗ_данные" (плоская таблица, перезаписывается целиком при
      каждом импорте - как "Данные_1С").
- [ ] `runAll()`: добавить `importDebtReport()` в общий пайплайн (try/catch как остальные
      импорты, не должен ронять весь прогон при сбое).
- [ ] Проверить на реальном файле образца (лежит в корне репо, не в git) - сверить, что
      топ-должники и суммы совпадают с ручным подсчётом в Python.

### Фаза 2. Backend: история и агрегация (~1.5ч)
- [ ] `saveDebtHistory()` - по аналогии с `saveFinancialHistory()`: раз в день снимок в
      новый лист "История_ДЗ" (дата, общая ДЗ, кол-во должников, возможно разбивка по
      менеджеру). Идемпотентно - если строка за сегодня уже есть, перезаписываем (отчёт
      от 1С раз в день, но наш `runAll()` гоняется чаще - см. новое расписание 6:05-21:05).
- [ ] `getDebtData(ss)` - агрегация для дашборда: `summary` (общая ДЗ, должников,
      просрочено > 30/60/90 дней), `by_customer` (контрагент, менеджер без телефона,
      долг, аванс, чистый баланс, дата последнего документа, дней с даты), `by_manager`
      (рулап на менеджера - для ответственности за взыскание).
- [ ] Подключить в `doGet()` - новое поле `debt: getDebtData(ss)` в основном payload
      (тот же лёгкий вес, что и остальные листы дашборда - без отдельного кэша, как
      Нормализованные_данные).

## Риски

- **Формула "чистой ДЗ" может быть не той, что нужна Владу** - начинаем с простой
  (долг минус аванс), явно помечено как открытый вопрос для уточнения после первого
  прогона на реальных данных.
- **1С может изменить формат отчёта** (структуру колонок/заголовков) - парсер по
  индексам колонок, а не по названиям листа - если 1С поменяет порядок колонок, разбор
  молча съедет. Стоит когда-нибудь добавить проверку заголовков (как в других импортах),
  не в этой итерации.
- **"Отдел" = "Коммерческий блок"** - подтверждено на выгрузке, что менеджеры там - те же
  трал-менеджеры (не отдельная категория), включаем наравне с остальными.

## Открытые вопросы

- Точные пороги "просрочки" (30/60/90 дней?) - уточнить с Владом при первом просмотре
  вкладки.
- Нужна ли колонка "Гарантийный депозит" в таблице должников как отдельный контекст,
  или Влад хочет её только в отдельном месте (связано с зарплатой менеджеров)?

**Уроки:**

  иерархии не пытаться пересчитать контрагентские итоги из документов, если сам отчёт

  (юрлицо), если бизнес-логика фактически размазана по нему (тендеры/переходный период) -
  фильтровать по тому измерению, которое реально соответствует ответственности (менеджер).
