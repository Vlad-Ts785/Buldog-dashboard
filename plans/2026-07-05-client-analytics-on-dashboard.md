## Цель

Перенести аналитику клиентской базы (топ-клиенты, win-back, растущие/снижающиеся, сезонность,
профиль менеджера) из разового локального Python-анализа в живой дашборд, чтобы она обновлялась
сама и не требовала ручного запуска скрипта.

## Контекст

Влад провёл со мной разовый анализ клиентской базы за 6.5 лет (2020-2026, файл "мега база.xlsx",
112 754 строки) в отдельном треке `.business/clients/` (см. `INDEX.md` там же) - через pandas
локально, результат в Excel. Теперь Влад хочет то же самое, но живое, на дашборде, и сохранил
исходник в отдельную Google Таблицу:
`https://docs.google.com/spreadsheets/d/1nXMXVxLiOK-7CXdSSFr7NcoCUvvyEDCjPyGhE0dv8es`.

Ключевое уточнение от Влада (2026-07-05): исторя в этой таблице - **разовая выгрузка по май 2026
включительно**. Июнь и июль 2026 уже приходят туда же, откуда их берёт сам дашборд - то есть не
нужно повторно грузить их вручную, нужно **сшивать** статичный исторический кусок с уже живыми
данными дашборда.

**Что нашлось при разведке живого скрипта** (`scripts/full_script_final.js`):

- Дашборд уже несколько недель ведёт СВОЙ конвейер заказов, почти один в один похожий на то, что
  я делал в Python:
  - Gmail → `Заказы_сырые` (`importOrdersReport()`, строка ~2157)
  - `Заказы_сырые` → `Заказы_данные` (`normalizeOrders()` + чистая функция `parseOrdersRawRows()`,
    строка ~2315) - уже фильтрует `TRAL_MANAGERS`/`TRAL_LOGISTS` (строка ~2012, 2021) и
    `INTERNAL_CLIENTS` (строка ~2006) - **списки совпадают 1-в-1** с теми, что я использовал в
    `analyze_clients.py` (кроме одного нюанса - см. "Хвост УМИАТ ЯРД" ниже).
  - Архивы по месяцам - листы `Заказы_YYYY-MM` (`ORDERS_ARCHIVE_PFX`), уже используются для
    "пропавших клиентов" (`computeLostCustomers_()`, строка ~2470) - по сути мини-версия win-back,
    уже на дашборде, просто по 2 месяцам, а не по всей истории.
  - Основной ID таблицы дашборда: `1jCPRXYDFcTpZIHdJfngZveOQFycu6qbcl-MoXBxtBRM` (в `CONFIG`,
    строка 135) - это ДРУГАЯ таблица, чем `мега база` Влада (`1nXMXVxL...`).

- **Важно:** формат колонок в `мега база.xlsx` НЕ совпадает 1-в-1 с форматом, который парсит
  `parseOrdersRawRows()` (например там `Начало` вместо `Начало работ`, нет колонок `Реализация`,
  `Оригинал получен`, `Адрес объекта`, `Единица измерения` называется `Ед. Изм` и т.д.) - похоже,
  историческая выгрузка 1С шла по чуть другому шаблону отчёта, чем текущий. **Нельзя** просто
  положить `мега база` как ещё один архивный лист `Заказы_YYYY-MM` и переиспользовать
  `parseOrdersRawRows()` без изменений - даты потеряются (`monthKey` не определится), часть строк
  тихо пропадёт.

## Решение (архитектура)

Не трогаем живой конвейер текущего месяца (`Заказы_сырые` → `Заказы_данные`, риск сломать то, что
уже работает и от чего зависит "Зарплата"/"Панель"). Вместо этого:

1. В таблице Влада (`1nXMXVxL...`) добавляем отдельную нормализацию под формат этой конкретной
   исторической выгрузки - выходной лист с той же СЕМАНТИКОЙ полей, что нужна для клиентской
   аналитики (Заказчик, Менеджер по продажам, Менеджер по снабжению, Тип техники, Сумма, Прибыль,
   Начало, признаки isInternal/isTralDept) - но не обязательно тем же порядком/именами колонок,
   что `Заказы_данные`. Константы (`INTERNAL_CLIENTS`, `TRAL_MANAGERS`, `TRAL_LOGISTS`) **не
   дублируем как новый список** - копируем 1-в-1 из `full_script_final.js`, с явным комментарием
   "источник истины - scripts/full_script_final.js, обновлять синхронно".
2. На дашборде (`full_script_final.js`, таблица `1jCPRXYDFcTpZIHdJfngZveOQFycu6qbcl-MoXBxtBRM`)
   добавляем новую функцию `getClientAnalyticsData_()`, которая:
   - читает нормализованный исторический лист из таблицы Влада
     (`SpreadsheetApp.openById('1nXMXVxL...')`, чтение по ID из другой таблицы - штатная
     возможность Apps Script, доп. прав не требует, если у скрипта есть доступ на чтение);
   - читает свежие месяцы из СВОЕЙ таблицы (`Заказы_данные` + архивы `Заказы_YYYY-MM`, начиная с
     июня 2026 - там, где обрывается историческая выгрузка);
   - склеивает оба куска в одну выборку по клиенту/менеджеру и считает то же самое, что считал
     `analyze_clients.py`: топ-клиенты, концентрация (Парето), сегменты по давности, win-back,
     растущие/снижающиеся, сезонность по месяцам, профиль менеджера.
3. Новый action в `doGet()`, например `action=client_analytics` (клиент целиком) и
   `action=manager_profile&manager=Цегельников` (как в примере дашборда для Славы) - JSON на
   выход, дашборд (`files/index.html`) рисует новую вкладку.

## Риски

- **Формат исторической выгрузки может ещё раз поменяться** (Влад уже дважды переоформлял
  экспорт - `База данных.xlsx` → `мега база.xlsx`). Нормализация в Фазе 1 должна быть терпима к
  дополнительным/переставленным колонкам (искать по имени, не по индексу - как уже сделано в
  `parseOrdersRawRows`/`findOrdersHeaderRowIndex_`), а не жёстко зашита под сегодняшний файл.
- **112 754 строки - Apps Script может упереться в лимит времени выполнения** (6 минут на
  вызов) при первой нормализации. Если не уложится - разбивать на батчи по годам.
- **Расхождение методик** - `computeLostCustomers_()` на дашборде уже считает "пропавших
  клиентов" по своей логике (15+ дней) - при добавлении полноценного win-back (60+ дней,
  историческая ценность) нужно явно развести эти два понятия в UI, не путать директора двумя
  похожими, но разными списками.
- **"Прибыль" не является чистой прибылью** (маржа по найму техники, см. переписку в этом чате) -
  если выводить эту колонку на дашборд, подписать явно, что это не чистая прибыль компании.

## Открытые вопросы (решено 2026-07-05)

- Повторные большие выгрузки в будущем? **Нет, разово, больше не придёт.** Значит
  автоматизировать регулярный повторный импорт истории не нужно - исторический кусок неизменен
  раз и навсегда (кроме этой одной загрузки).
- УМИАТ ЯРД ООО (НАШ) - **оставить как есть, не трогать `INTERNAL_CLIENTS` в рамках этого плана.**
  Важное следствие: нормализация в Фазе 1 будет копировать `INTERNAL_CLIENTS` 1-в-1 с живого
  дашборда (без моего собственного доп. фильтра `(НАШ)`, который я использовал в разовом
  Python-анализе) - то есть 2857 строк УМИАТ ЯРД **попадут** в новую историческую аналитику
  как если бы это был обычный клиент, пока список не поправят отдельно. Логирую это явно при
  нормализации (строка/сумма), чтобы не потерялось.
- Где вкладка на дашборде - **отдельная вкладка верхнего уровня** ("Клиенты"), не подраздел
  внутри "По менеджерам".
