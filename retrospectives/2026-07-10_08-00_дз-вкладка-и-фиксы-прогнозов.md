# Ретро: вкладка ДЗ (дебиторка) + фиксы прогнозов и триггера

**Дата:** 2026-07-07 - 2026-07-10 (один длинный чат, несколько дней подряд)
**Длительность:** много часов, несколько десятков коммитов
**План:** `plans/2026-07-08-debt-receivables-tab.md` (главный, самый актуальный срез там)

## 1. Задача

Началось с мелочи (время триггера `runAll()` под расписание 1С), выросло в большую
работу над точностью прогнозов выручки, а основную часть времени заняла новая вкладка
"ДЗ" (дебиторская задолженность) - реальные данные из отчёта 1С вместо приблизительной
оценки "должники" на других страницах дашборда.

## 2. Как решал

Каждая фича - коммит, `git fetch/merge origin/main` перед пушем (параллельный чат
регулярно пушит одновременно - конфликтов не было ни разу, авто-мерж), `clasp push` +
`clasp create-version` + `clasp redeploy` вместо ручного деплоя через браузер (Влад:
"сам сделай деплой, ты всё знаешь" - с этого момента деплою сам без вопроса).

Для ДЗ: сначала попросил у Влада реальный образец файла (не гадал по описанию), разобрал
структуру через Python/openpyxl (кириллица в путях Windows через bash - работает только
через `PYTHONIOENCODING=utf-8` + `os.chdir` внутри Python, не через bash cd напрямую).
Логику парсинга проверял в Node.js на РЕАЛЬНЫХ данных на каждом шаге (не только моками) -
это выявило минимум 2 настоящих бага до того, как Влад их нашёл сам (сумма документов
vs сумма контрагента; баланс после фикса кросс-юрлицного сведения).

Большая часть работы - не первая реализация, а серия точных правок по фидбэку Влада:
он тестировал вживую и присылал конкретные наблюдения ("вместо 210 тысяч видим меньше",
"88 дней - откуда цифра", "сумма корзин меньше общей ДЗ"). Каждый раз - сначала
диагностировал ПРИЧИНУ на реальных данных, потом чинил, а не гадал.

## 3. Решил?

- [x] Да, полностью - все запрошенные фичи внедрены и задеплоены (версия Apps Script 98
      на конец сессии). Не проверено ТОЛЬКО то, что физически нельзя проверить без
      доступа к почте Влада (живой Gmail-импорт) - помечено в плане явно.

## 4. Что можно было лучше

- **Первая версия формулы баланса ДЗ была неверной** (долг минус аванс по ВСЕМ юрлицам
  сразу, а не по каждому отдельно) - можно было сразу спросить у Влада "а если у клиента
  долг на одном юрлице и аванс на другом - это законно сводить?" вместо того, чтобы ждать,
  пока он сам это заметит на реальных цифрах. Урок: при первой реализации агрегации
  "клиент через несколько сущностей" - явно проговорить сценарий смешанных знаков, не
  ждать, что тестировщик его найдёт.
- Первая версия плана предполагала фильтр TRAL_MANAGERS "для подстраховки" - это
  выглядело безопасным решением, но оказалось потенциальным источником тихой потери
  данных, когда Влад расширил список менеджеров в самом 1С. Урок (уже записан в плане):
  не дублировать фильтр, если источник ИМ ЖЕ управляется пользователем напрямую.
- Несколько раз забывал, что правки в файле НЕ подхватываются уже открытой в preview
  страницей без `window.location.reload()` - терял время на "почему не сработало", хотя
  код был правильный.

## 5. Как было / как стало

- **Было:** триггер `runAll()` не совпадал с расписанием 1С (0-4ч задержка); прогноз
  выручки в разных местах дашборда считался по-разному и расходился; "должники" на
  дашборде - грубая оценка (`заказ - оплата`) без реальных данных дебиторки; телефон
  менеджера иногда протекал в UI.
- **Стало:** триггер синхронизирован с 1С (+5 мин запас); единая функция `calcPaceRatio_`
  + единый числитель "по вчера" для всех прогнозов выручки; сравнение выручки с прошлым
  месяцем/годом на Панели; новая вкладка "ДЗ" с реальными данными из 1С (129→139 реальных
  должников на ~35М ₽, разбивка по 5 юрлицам группы, история по дням со стрелками
  тенденций, детальная структура долга по клику, фильтр по менеджеру); телефон нигде не
  протекает.
